# Assumptions

The paper does not fully specify every engineering detail needed for a runnable codebase. This minimal reproduction uses the following assumptions.

1. Initial pursuer coordinates are a straight line along the x-axis, centered around zero. The paper states that pursuers are initially aligned in a straight line but does not provide exact coordinates. Source: Section II-A.
2. The target starts at a fixed configurable point in front of pursuers for deterministic smoke tests. The paper states that the target is randomly positioned in a circular area ahead of pursuers but does not provide the circle parameters. Source: Section II-A.
3. Obstacles are loaded from `config.yaml`. The paper defines circular obstacles but does not provide the full numeric obstacle distribution used in all experiments. Source: Section II-A and Section V.
4. Collision penalty `R_l` is `-3` only when a collision or safety-buffer violation occurs; otherwise it is `0`. The paper gives `R_l = -3` as the penalty for collisions but does not explicitly state the no-collision value. Source: Section IV-C.1, Eq. (18).
5. Local observation derivative terms `dot_Q_iT` and `dot_omega` are set to zero in the minimal implementation. The paper names these terms but does not specify the discrete estimator. Source: Section IV-C.2.
6. The centralized critic global state concatenation is simplified. The paper says the critic evaluates the overall state but does not give the exact tensor layout. Source: Section IV-C.3.
7. PPO uses mean team reward with a centralized scalar value. The paper gives the MAPPO/rMAPPO framework but does not provide all implementation details for value targets and advantage batching. Source: Section III-A, Section IV-C.3, Algorithm 1.
8. Guide-policy actions are stored with log probabilities evaluated under the current actor for PPO updates. The paper describes mixed guide/learned rollouts but does not specify off-policy correction for guide-generated actions. Source: Section IV-A and Algorithm 1.
9. Target policy defaults to static in the configuration. PFP and GP are implemented from Eqs. (20)-(21). AP and MCP require Apollonius/Voronoi geometry and remain future work. Source: Section V-D, Eqs. (20)-(23).
10. The vectorized server trainer can advance the JSRL and curriculum regulators by update count via `vector_training.curriculum_progress: updates`. The paper defines these schedules by episode count, but one vector update contains many parallel episodes; using raw completed episode count made Eqs. (11)-(14) decay before the actor had enough autonomous training data.
11. `config_vector_server.yaml` sets `regulators.cg = 0.9` for the vectorized server run so guide-policy handoff overlaps the obstacle-radius curriculum. This is a training-stability assumption, not a paper-reported hyperparameter.
12. The vector trainer periodically evaluates deterministic actor-only execution and saves `mra_rlec_best.pt`. This checkpoint selection rule is an engineering addition because guide-assisted rollout success can overestimate the learned policy.
13. The user-provided `environment.py` is now treated as the higher-priority engineering reference for guide-policy details. Because that file depends on an external MPE/onpolicy/world callback stack, its `MultiAgentEnv` wrapper is not imported directly. Instead, its `policy_u`, `limit_action_inf_norm`, and linear `set_JS_curriculum` behavior are translated into the current standalone planar implementation under `guide_policy.mode: environment_py` and `regulators.jsrl_mode: environment_py`.
14. Obstacle safety buffers are scaled with obstacle radius during the obstacle curriculum. The paper only defines radius growth in Eq. (14); this codebase adds buffers for collision checking, so leaving buffers nonzero when radius is zero would keep point obstacles active during the supposedly obstacle-free curriculum phase.
15. A conservative `guide_policy.safety_filter` is enabled for `environment_py` guide mode. The uploaded `environment.py` force law was designed for a different MPE/onpolicy dynamics stack; when translated directly into this second-order circular-agent environment it collided within about 25 steps in the easiest curriculum stage. The safety filter only adds short-range pair/obstacle avoidance around the environment.py nominal action.
16. For vector server training, `regulators.rho0` is reduced to `0.4` so the learned actor controls the tail of each episode early in training. With `rho0=0.8`, guide-only episodes reached success before the actor ever acted, producing `actor_loss=0` and actor-only success remaining zero.
17. Server vector training uses `num_envs=16`, `rollout_length=64`, and one PyTorch CPU thread per process. The current planar environment and safety layer are Python/CPU bound, so the previous `64 x 128` rollout setting overloaded CPU sampling and delayed the first logged update for tens of minutes.
18. `config_vector_server.yaml` now uses `training.policy_mode: residual`. The active branch in the uploaded `environment.py` performs a hard handoff from `policy_output` to `network_output`; the server logs showed this hard handoff consistently collapsed to early collisions. The residual mode is an engineering stabilization: `environment_py` guide actions remain the nominal controller, while the actor learns a bounded residual correction. This keeps the reproduction focused on encirclement behavior in the simplified planar-circle environment rather than forcing a brittle direct-action transfer.
19. In residual mode, behavior-cloning pretraining targets a zero residual instead of the guide action, and final checkpoint evaluation also composes `guide_action + residual_scale * actor_action`. Therefore old `checkpoints/pretrain_actor.pt` files trained under direct-action semantics must be regenerated before server training.
20. Residual-mode pretraining writes `actor.log_std = -2.0` by default. This is an engineering stabilization so the first PPO rollouts remain close to the guide policy; without it, a zero-mean actor can still sample large random residuals.
21. The server configuration adds dense planar encirclement shaping terms `reward.angle_shaping_weight` and `reward.distance_shaping_weight`, plus a terminal timeout penalty. This is not an additional paper formula; it is an engineering response to the simplified planar-circle setting where the translated `environment.py` guide safely approaches the target but times out with nonzero angular spacing error under the paper's strict Eq. (2) success test.
22. Vector training checkpoint selection now breaks ties by timeout rate and mean evaluation reward when all evaluated checkpoints have the same success and collision rate. This prevents the first zero-success checkpoint from remaining `mra_rlec_best.pt` for the whole run.
23. The `environment_py` guide can optionally add `formation_correction`, a dense slot-based ring controller that preserves the current cyclic order of pursuers and drives them toward evenly spaced points on the capture circle. This is an engineering addition for the simplified planar-circle reproduction: the uploaded `environment.py::policy_u` has target attraction, inter-agent repulsion, obstacle repulsion and damping, but no explicit angular slot-assignment term, which caused safe timeouts with final angle error around `0.64` to `0.68` radians. Formation correction is delayed, ramped, obstacle-gated, and uses an obstacle-aware ring orientation search because an immediate straight slot controller collided with obstacles at the final curriculum stage.
24. In the server residual configuration, `residual_scale` is reduced to `0.1` and `entropy_coef` to `0.0002`. The baseline guide is expected to solve the planar encirclement task in the corrected feasible environment, so PPO residual exploration should make small corrections rather than override the guide controller.
25. The server planar environment now sets `target_initial_position` to `[0.0, 3.5]`. The previous engineering assumption `[0.0, 2.2]` placed the target capture circle almost tangent to the assumed obstacle safety zones; guide diagnostics at the final curriculum stage either collided with obstacles or timed out with agents held outside the capture ring. Since the paper does not provide the exact target-position distribution or obstacle coordinates, the simplified planar setting must use a geometrically feasible target/obstacle layout before RL training is meaningful.
